## Objective

Get the Search runtime to the best long-term architecture:

- JS prepares stable transaction inputs and structural source universes
- UI/native executes visible transitions and render-local decisions
- transaction ownership is explicit
- motion pressure is runtime-owned and transaction-aware
- no long-lived compatibility projection or split-owner hot path remains

## Current Reality

What is already materially landed:

- prepared results snapshots exist
- profile open/close has a prepared command model
- restaurant profile visible content is native-hosted
- the mounted overlay host is now app-level, not Search-local, and the overlay store is route-entry based without the old key-only compatibility mirrors
- sticky label source projection is candidate-static on the hot path
- native owns sticky observation cadence and sticky memory
- motion pressure is now runtime-instance-owned at the Search root, and old process-global singleton policy is gone

## Target Ideal Shape

### 1. Prepared presentation is the only visible-lane owner

- prepared results/profile transactions are committed once
- visible execution reads one transaction model
- compatibility reveal/dismiss phase enums do not drive downstream behavior
- local JS pending-intent state does not sit beside the transaction owner

### 2. Profile presentation is executor-first at an explicit split boundary

- JS owns shell/app policy: shell-state application, route intent, hydration policy, close-time app policy, and Search-specific foreground/clear commands
- native owns frame-critical camera/sheet command transport plus completion binding/execution
- prepared profile transactions still execute through one ordered runtime-owned phase loop, and future slices should delete wrapper seams around this split instead of promoting the JS shell lane into native

### 3. Native restaurant host is a stronger bridge boundary

- typed/native payload channel instead of `snapshotJSON`
- action channel remains explicit, but transport is not generic string serialization
- visible lane stays native-hosted

### 4. Motion pressure is runtime-owned

- no module-global singleton policy
- state is instance-scoped and explicit
- no implicit `Date.now()` inside admission policy
- transaction-awareness and fairness remain

### 5. Sticky labels are fully executor-shaped

- JS emits the static candidate universe
- native owns live preferred-side choice and memory
- preferred-side changes do not require JS source rewrites
- native preferred-side admission is driven through retained feature-state, not source-property rewrites

### 6. Delete gates are real

- when a concern is promoted, old compatibility plumbing is removed in the same slice
- stale diagnostics, compatibility comments, and dead bridge surfaces are deleted
